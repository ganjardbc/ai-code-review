# Glossary

Alphabetical definitions of terms used throughout the AI Code Reviewer documentation.

---

## 9Router

The AI gateway service used by this application. 9Router acts as a proxy and router between the application and underlying large language models (LLMs). The application sends code review prompts to 9Router via its API, and 9Router routes the request to the configured model (e.g., GPT-4o). Configured via `NINE_ROUTER_API_KEY` and `NINE_ROUTER_BASE_URL`.

---

## AOF (Append-Only File)

A Redis persistence mode where every write operation is logged to a file (`appendonly.aof`). When Redis restarts, it replays the AOF to reconstruct the dataset. AOF provides better durability than RDB snapshots but uses more disk space. Recommended for production deployments.

---

## BullMQ

An open-source Node.js job queue library backed by Redis. BullMQ handles job serialization, distributed locking (preventing two workers from processing the same job), retry scheduling, and job state tracking (waiting, active, completed, failed). The AI Code Reviewer uses BullMQ for the review job queue.

---

## Clean Architecture

A software design pattern that separates code into concentric layers: domain (innermost), application, infrastructure, and presentation (outermost). Dependencies point inward only — outer layers depend on inner layers, never the reverse. In this project, the domain layer contains only interfaces; infrastructure implements them.

---

## diff

A text representation of changes between two versions of a file. The application uses the **unified diff** format, which shows removed lines prefixed with `-` and added lines prefixed with `+`. Diffs are generated by git and sent to the AI provider for review.

---

## HMAC (Hash-based Message Authentication Code)

A cryptographic technique for verifying both the integrity and authenticity of a message. GitHub signs webhook payloads with HMAC-SHA256 using a shared secret (`GITHUB_WEBHOOK_SECRET`). The application recomputes the HMAC from the raw request body and secret, then compares it to the signature in the `X-Hub-Signature-256` header. A mismatch indicates tampering or a misconfigured secret.

---

## inline comment

A code review comment attached to a specific line of a specific file in a pull request or merge request. The AI reviewer generates inline comments when it identifies issues at a particular line. If the suggested line number is invalid (not part of the diff), the application falls back to a general comment on the PR/MR body.

---

## IVcsClient

The domain interface (`src/domain/interfaces/IVcsClient.ts`) that all version control system (VCS) provider integrations must implement. Defines methods for fetching diffs and posting review comments. Current implementations: `GithubService` and `GitlabService`. Consumers (use cases) depend on `IVcsClient`, not on the concrete implementation.

---

## IAiProvider

The domain interface (`src/domain/interfaces/IAiProvider.ts`) that all AI provider integrations must implement. Defines a single `review(prompt: string, diff: string): Promise<ReviewResult>` method. Current implementation: `NineRouterService`. New AI providers (OpenAI direct, Anthropic, etc.) implement this interface.

---

## job

A unit of work in the BullMQ queue. When a webhook is received, the API server creates a job containing the PR/MR details (repository URL, PR number, commit SHAs, VCS provider). The worker picks up the job and runs the full review pipeline. Jobs have states: waiting, active, completed, failed, delayed.

---

## merge request (MR)

GitLab's term for a code change proposal — equivalent to GitHub's "pull request". A merge request asks to merge one branch into another. The AI reviewer subscribes to merge request events via GitLab webhooks.

---

## OpenCode

The AI-powered code review component. OpenCode refers to the AI model interaction layer — it receives the formatted code diff and system prompt, and returns structured review comments. In this implementation, OpenCode functionality is provided through 9Router.

---

## pull request (PR)

GitHub's term for a code change proposal — equivalent to GitLab's "merge request". A pull request asks to merge one branch into another. The AI reviewer subscribes to pull request events via GitHub webhooks.

---

## RDB (Redis Database Backup)

A point-in-time snapshot of the Redis dataset, saved to `dump.rdb`. Redis creates RDB files automatically based on the `save` configuration. RDB snapshots are compact and fast to load, but may miss recent writes (between the last snapshot and a crash). Used for backup and restore.

---

## severity

A label attached to each AI review comment indicating the urgency of the finding:

| Level | Meaning |
|-------|---------|
| `info` | Informational — style suggestion, improvement opportunity |
| `warning` | Potential problem — code works but may cause issues |
| `error` | Definite problem — bug, security issue, data corruption risk |

---

## unified diff

The standard text format for representing code changes. Each changed file is represented with `---` (old file) and `+++` (new file) headers, followed by hunks prefixed with `@@` markers showing line ranges. Changed lines are prefixed with `-` (removed) or `+` (added). Context lines (unchanged) have no prefix.

Example:

```
--- a/src/index.ts
+++ b/src/index.ts
@@ -10,7 +10,8 @@
 function greet(name: string) {
-  return 'Hello ' + name;
+  return `Hello ${name}`;
 }
```

---

## webhook

An HTTP callback mechanism used by GitHub and GitLab to notify external services of repository events. When a pull request is opened, GitHub sends a `POST` request to the configured webhook URL with a JSON payload describing the event. The AI Code Reviewer registers as a webhook target to receive these notifications and trigger reviews.

---

## workspace

A temporary directory created for each review job at `WORKSPACE_DIR/job-<uuid>`. The repository is cloned into this directory, the diff is generated, and the directory is deleted after the job completes (in a `finally` block). Workspace directories should not persist after jobs finish. See [Workspace Cleanup](../operations/workspace-cleanup.md).

---

## worker

The background process that consumes review jobs from the BullMQ queue. Workers are independent from the API server — they are started with `node dist/worker.js` (or `pnpm start:worker`). Multiple workers can run in parallel, all consuming from the same Redis-backed queue. Workers perform the actual review: git clone, diff extraction, AI call, and comment posting.
